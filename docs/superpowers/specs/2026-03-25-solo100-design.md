**日期**: 2026-03-25
**状态**: 草稿

## 1. 项目定位

solo100 是一个帮助工程师在 AI 时代提效 100 倍的工具平台。核心思路是：单人开发者借助标准 Git Flow + 完整开发工作流，通过多 AI Agent 并行执行，模拟多人团队协作的开发速度。

---

## 2. 核心流程

系统包含两个解耦的独立功能：**需求管理** 和 **Feature 开发流程**。

### 2.1 需求管理

将大需求拆解成 Feature 列表，与 Feature 开发流程无直接依赖。

```
需求输入 (Web UI / 飞书)
    ↓
FeaturePlanner → AI 拆解成 Feature 列表（每个 Feature 是独立功能单元）
    ↓
[人工审查] 调整 Feature 列表（增删改、调整优先级）
    ↓
确认后 Feature 进入待开发状态，人工按需启动
```

### 2.2 Feature 开发流程

Feature 可以来自需求管理，也可以人工直接创建。多个 Feature 可并行执行，每个 Feature 独立走完整开发流程。

**Feature 状态机：**

```
pending（待开发）
    ↓ 人工启动
brainstorming（AI 分析需求）
    ↓ [人工介入 1] 确认 brainstorming 结果
planning（AI 生成实现计划）
    ↓ [人工介入 2] 确认实现计划
implementing（Agent 执行代码）
    ↓
testing（执行测试 → 生成测试报告）
    ↓ [人工介入 3] 确认测试结果
    ↓ 通过              ↓ 失败：人工选择"回到 brainstorming 重试"或"忽略继续"
reviewing（AI code review）
    ↓ [人工介入 4] Approve / Reject
    ↓ Approve           ↓ Reject：人工选择"回到 brainstorming 重试"
approved（等待 verification）
    ↓
verifying
    ↓ 通过              ↓ 失败：回退到 approved，人工选择"重新触发"或"回到 brainstorming"
merged（PR Merge → 清理 worktree）
```

**可回溯节点**：testing 失败、review reject、verification 失败时，人工均可选择回到 `brainstorming` 重新开始，`retry_count` 递增。回溯是增量的：brainstorming 基于上次结果 + 失败原因重新分析，planning 基于上次 plan 修订，implementing 在现有代码上修改，不重置 branch 或清空已有工作。

**failed 终态**：`retry_count` 超过 `max_retries`（默认 3）仍未通过，进入 `failed` 终态。人工可选择"重置重试"（`retry_count` 清零，回到 `brainstorming`）或"放弃"（进入 `archived` 终态）。

### 2.3 verification 阶段

verification 是 Approve 后、merge 前的最终验证：
1. Feature branch rebase 到最新主分支，检查 merge conflict
2. 运行完整测试套件（单元测试 + 集成测试）
3. 全部通过后自动触发 PR merge，清理 worktree

verification 失败时，Feature 回退到 `approved` 状态，通知人工介入。

## 3. 模块设计

| 模块 | 职责 |
|------|------|
| **FeaturePlanner** | 接收需求，调用 LLM 拆解成 Feature 列表，与 Feature 开发流程解耦 |
| **Feature Executor** | 驱动单个 Feature 的完整开发流程状态机，协调各阶段执行 |
| **Agent Executor** | 管理 Agent 生命周期，抽象 BaseAgent 接口，支持多种 Agent 实现 |
| **Git Manager** | 按需 clone 仓库，管理 feature branch、commit、PR 创建、rebase、merge、worktree 清理 |
| **Review Engine** | 对 Feature 产出的代码做 AI code review，生成结构化报告 |
| **Notification Hub** | 统一推送事件到 Web UI（WebSocket）和飞书（Bot API） |
| **Approval Gateway** | 处理所有人工介入节点的 approve/reject，驱动 Feature 状态机流转 |

## 4. Agent 适配层

```python
class BaseAgent:
    async def brainstorm(self, feature: Feature, previous: BrainstormResult | None = None, failure_reason: str | None = None) -> BrainstormResult: ...
    async def plan(self, feature: Feature, brainstorm: BrainstormResult, previous_plan: Plan | None = None) -> Plan: ...
    async def implement(self, feature: Feature, plan: Plan, worktree_path: str) -> ImplementResult: ...

class ClaudeCodeAgent(BaseAgent): ...
class CodexAgent(BaseAgent): ...
```

- 全局配置默认 Agent 类型
- 后续可按 Project 或 Task 级别覆盖配置

## 5. 数据模型

```
Project
  - id, name, ssh_url, default_branch, default_agent_id, ssh_key_ref, created_at
  # ssh_key_ref: 引用 Secret Manager 中的 SSH 私钥，不存明文

Requirement
  - id, project_id, title, description, created_at
  # 大需求，FeaturePlanner 的输入

Feature
  - id, project_id, requirement_id (nullable), title, description
  - status: pending | brainstorming | planning | implementing | testing | reviewing | approved | verifying | merged | failed | archived
    # pending: 待开发，等待人工启动
    # approved: 人工 approve 后等待 verification；verification 失败回退到 approved
    # archived: 人工放弃的终态
  - branch（系统自动生成：feat/<feature-id>-<title-slug>）
  - pr_url, worktree_path
  - retry_count, max_retries（默认 3）
  - created_at

FeatureExecution
  - id, feature_id, attempt_number, stage, result_json, created_at
  # stage 枚举：brainstorming | planning | implementing | testing | reviewing | verifying
  # 只记录有 AI/Agent 执行动作的阶段；pending/approved/merged/archived 不产生执行记录
  # 重试时 attempt_number 递增，同一 attempt 内每个 stage 一条记录

AgentConfig
  - id, name, type (claude_code | codex | ...), api_key_ref, is_default
  # api_key_ref: 引用 Secret Manager，不存明文

ReviewReport
  - id, feature_id, attempt_number
  - ai_summary, ai_issues[]           # AI code review 的结果
  - human_decision: approve | reject  # 对应人工 Approve/Reject 的决定
  - decided_by, decided_at
```

## 6. 技术栈

| 层 | 技术 |

## 7. 双入口设计（Web UI / 飞书）

Web UI 和飞书功能对等，均支持：
- 创建需求，查看 Feature 列表
- 启动 Feature 开发流程
- 每个人工介入节点的 Approve / Reject 操作
- 实时查看 Feature 执行进度和日志
- 查看测试报告、code review 报告
- 飞书通过消息卡片支持所有操作，同时推送状态变更通知

---

## 8. Git 工作流

- 每个 Feature 对应一个独立 branch，命名规则：`feat/<feature-id>-<title-slug>`，系统自动生成
- Agent 执行时按需 clone 到临时 worktree 目录
- 多个 Feature 并行时各自在独立 branch 开发，互不干扰
- merge 前自动 rebase 到最新主分支；rebase 冲突时暂停并通知人工解决
- 主分支保护，所有变更必须经过 PR + 人工 Approve + verification 通过
