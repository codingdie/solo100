# solo100 v0.5 详细设计文档

**日期**: 2026-03-26
**状态**: 草稿
**基于版本**: v0.4

---

## 1. 版本目标

v0.5 在 v0.4（飞书双向集成）基础上，完成 Agent 执行层的横向扩展。核心目标：

- 将 v0.1 硬编码的 Claude Code Agent 抽象为可替换的 Agent 适配层
- 新增 Codex Agent 作为第一个非 Claude 实现，验证适配层设计
- 支持全局默认 Agent 配置，以及按 Project 级别覆盖
- 提供 AgentConfig 管理界面（Web UI + 飞书）
- 记录 Agent 执行日志，支持多 Agent 性能对比

**交付物**：用户可以在系统中配置多个 Agent，选择不同 Agent 执行 Feature 开发流程，并通过日志对比各 Agent 的执行效果。

---

## 2. 新增功能范围（相对 v0.4 的 delta）

v0.1~v0.4 的所有功能保持不变，v0.5 仅在以下方面新增或变更：

| 类别 | 变更内容 |
|------|----------|
| Agent 适配层 | 引入 `BaseAgent` 抽象接口，重构 `ClaudeCodeAgent` 实现该接口，新增 `CodexAgent` |
| 数据模型 | `AgentConfig` 新增字段；`Project` 新增 `default_agent_id`；`FeatureExecution` 新增 `agent_id`、`duration_ms` |
| 全局配置 | 系统级默认 Agent 配置项 |
| Project 配置 | Project 创建/编辑时可指定 `default_agent_id`，覆盖全局默认 |
| Web UI | AgentConfig 管理页面；Project 编辑页新增 Agent 选择；Feature 执行日志新增 Agent 信息和耗时 |
| 飞书 | AgentConfig 查询/切换指令；Feature 执行通知新增 Agent 名称和耗时 |
| 执行日志 | `FeatureExecution` 记录每阶段使用的 Agent、耗时、token 消耗（如 Agent 支持） |

**不变更的内容**：Feature 状态机、Git 工作流、Review Engine、Notification Hub、Approval Gateway、FeaturePlanner 均无变化。

## 3. Agent 适配层详细设计

### 3.1 接口定义

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

# --- 输入/输出数据结构 ---

@dataclass
class Feature:
    id: str
    title: str
    description: str
    project_id: str
    branch: str
    worktree_path: str

@dataclass
class BrainstormResult:
    summary: str           # 需求理解摘要
    key_points: list[str]  # 关键实现要点
    risks: list[str]       # 潜在风险和注意事项
    raw_output: str        # Agent 原始输出，用于日志

@dataclass
class Plan:
    steps: list[PlanStep]  # 有序实现步骤
    estimated_files: list[str]  # 预计涉及的文件
    raw_output: str

@dataclass
class PlanStep:
    order: int
    description: str
    file_path: Optional[str]

@dataclass
class ImplementResult:
    success: bool
    changed_files: list[str]  # 实际修改的文件列表
    commit_hash: Optional[str]
    error_message: Optional[str]  # 失败时的错误信息
    raw_output: str

# --- 抽象基类 ---

class BaseAgent(ABC):
    """
    所有 Agent 实现必须继承此类。
    每个方法对应 Feature 开发流程中的一个阶段。
    """

    def __init__(self, config: "AgentConfig"):
        self.config = config

    @abstractmethod
    async def brainstorm(self, feature: Feature, previous: BrainstormResult | None = None, failure_reason: str | None = None) -> BrainstormResult:
        """
        分析 Feature 需求，输出理解摘要和关键要点。
        在 brainstorming 阶段调用。
        previous: 上一次 brainstorm 结果（回溯时传入）
        failure_reason: 上一次失败或被拒绝的原因（回溯时传入）
        """
        ...

    @abstractmethod
    async def plan(self, feature: Feature, brainstorm: BrainstormResult, previous_plan: Plan | None = None) -> Plan:
        """
        基于 brainstorm 结果生成具体实现计划。
        在 planning 阶段调用。
        previous_plan: 上一次 plan 结果（回溯时传入）
        """
        ...

    @abstractmethod
    async def implement(self, feature: Feature, plan: Plan, worktree_path: str) -> ImplementResult:
        """
        在 worktree_path 中执行代码实现。
        Agent 负责在 worktree 内完成代码修改和 commit。
        在 implementing 阶段调用。
        """
        ...

    def name(self) -> str:
        """返回 Agent 显示名称，默认取 config.name"""
        return self.config.name
```

### 3.2 ClaudeCodeAgent 实现

ClaudeCodeAgent 是 v0.1 已有实现的重构版本，行为不变，仅适配 BaseAgent 接口。

```python
class ClaudeCodeAgent(BaseAgent):
    """
    通过 claude CLI 工具执行 Agent 任务。
    brainstorm/plan 阶段通过 claude -p 传入 prompt 获取结构化输出。
    implement 阶段通过 claude --dangerously-skip-permissions 在 worktree 内执行。
    """

    def brainstorm(self, feature: Feature) -> BrainstormResult:
        prompt = self._build_brainstorm_prompt(feature)
        raw = self._run_claude(prompt, cwd=feature.worktree_path)
        return self._parse_brainstorm(raw)

    def plan(self, feature: Feature, brainstorm: BrainstormResult) -> Plan:
        prompt = self._build_plan_prompt(feature, brainstorm)
        raw = self._run_claude(prompt, cwd=feature.worktree_path)
        return self._parse_plan(raw)

    def implement(self, feature: Feature, plan: Plan) -> ImplementResult:
        prompt = self._build_implement_prompt(feature, plan)
        raw = self._run_claude_implement(prompt, cwd=feature.worktree_path)
        return self._parse_implement_result(raw, feature.worktree_path)

    def _run_claude(self, prompt: str, cwd: str) -> str:
        # 调用 claude CLI，返回原始输出
        ...

    def _run_claude_implement(self, prompt: str, cwd: str) -> str:
        # 调用 claude --dangerously-skip-permissions，允许文件操作
        ...
```

**关键差异**：ClaudeCodeAgent 的 implement 阶段直接在 worktree 内操作文件系统，由 claude CLI 完成 commit。

### 3.3 CodexAgent 实现

CodexAgent 通过 OpenAI Codex API（或 `codex` CLI）执行任务。

```python
class CodexAgent(BaseAgent):
    """
    通过 OpenAI Codex CLI 执行 Agent 任务。
    brainstorm/plan 阶段调用 Codex API 获取结构化输出。
    implement 阶段通过 codex CLI 在 worktree 内执行代码修改。
    """

    def brainstorm(self, feature: Feature) -> BrainstormResult:
        prompt = self._build_brainstorm_prompt(feature)
        raw = self._call_codex_api(prompt)
        return self._parse_brainstorm(raw)

    def plan(self, feature: Feature, brainstorm: BrainstormResult) -> Plan:
        prompt = self._build_plan_prompt(feature, brainstorm)
        raw = self._call_codex_api(prompt)
        return self._parse_plan(raw)

    def implement(self, feature: Feature, plan: Plan) -> ImplementResult:
        prompt = self._build_implement_prompt(feature, plan)
        raw = self._run_codex_cli(prompt, cwd=feature.worktree_path)
        return self._parse_implement_result(raw, feature.worktree_path)

    def _call_codex_api(self, prompt: str) -> str:
        # 调用 OpenAI API，model=codex 或 gpt-4o
        ...

    def _run_codex_cli(self, prompt: str, cwd: str) -> str:
        # 调用 codex CLI，在 worktree 内执行
        ...
```

### 3.4 各 Agent 实现差异对比

| 维度 | ClaudeCodeAgent | CodexAgent |
|------|----------------|------------|
| 执行方式 | claude CLI（本地进程） | codex CLI / OpenAI API |
| brainstorm/plan | claude -p + prompt | OpenAI Chat API |
| implement | claude --dangerously-skip-permissions | codex CLI |
| commit 方式 | claude CLI 自动 commit | Agent 执行后系统自动 commit |
| API Key 配置 | ANTHROPIC_API_KEY | OPENAI_API_KEY |
| token 消耗记录 | 从 claude 输出解析 | 从 API response 读取 |
| 并发限制 | 受 claude CLI 并发限制 | 受 OpenAI API rate limit |

### 3.5 Agent 执行器（AgentExecutor）

AgentExecutor 负责：
1. 根据 Feature 所属 Project 的 `default_agent_id`（或全局默认）实例化对应 Agent
2. 调用 Agent 方法，记录耗时和结果到 `FeatureExecution`
3. 捕获异常，统一转换为 `ImplementResult(success=False, ...)`

```python
class AgentExecutor:
    def __init__(self, agent_config_repo: AgentConfigRepository):
        self.agent_config_repo = agent_config_repo

    def get_agent(self, project: Project) -> BaseAgent:
        # 优先取 project.default_agent_id，否则取全局默认
        agent_id = project.default_agent_id or self._get_global_default_agent_id()
        config = self.agent_config_repo.get(agent_id)
        return self._instantiate(config)

    def _instantiate(self, config: AgentConfig) -> BaseAgent:
        registry = {
            "claude_code": ClaudeCodeAgent,
            "codex": CodexAgent,
        }
        cls = registry.get(config.type)
        if not cls:
            raise ValueError(f"未知 Agent 类型: {config.type}")
        return cls(config)

    def run_brainstorm(self, feature: Feature, project: Project, last_execution_result: BrainstormResult | None = None, failure_reason: str | None = None) -> BrainstormResult:
        agent = self.get_agent(project)
        start = time.monotonic()
        try:
            result = await agent.brainstorm(feature, previous=last_execution_result, failure_reason=failure_reason)
        finally:
            duration_ms = int((time.monotonic() - start) * 1000)
            self._record_execution(feature, "brainstorming", agent.config.id, duration_ms, result)
        return result

    # run_plan / run_implement 同理
```

## 4. AgentConfig 数据模型变更

### 4.1 AgentConfig 表变更

在 v0.4 基础上，`AgentConfig` 新增以下字段：

```
AgentConfig
  - id            string, PK
  - name          string          # 显示名称，如 "Claude Code (默认)"
  - type          string          # 枚举：claude_code | codex
  - api_key_ref   string          # 引用 Secret Manager，不存明文
  - is_default    bool            # 是否为全局默认 Agent
  - extra_config  json (nullable) # Agent 特有配置，如 model 名称、temperature
  - created_at    datetime
  - updated_at    datetime        # 新增：记录最后修改时间
```

`extra_config` 示例：

```json
// ClaudeCodeAgent
{
  "model": "claude-sonnet-4-5",
  "max_tokens": 8192
}

// CodexAgent
{
  "model": "gpt-4o",
  "temperature": 0.2
}
```

### 4.2 Project 表变更

`Project` 已在主设计文档中包含 `default_agent_id` 字段，v0.5 正式启用该字段：

```
Project
  - default_agent_id  string (nullable, FK -> AgentConfig.id)
  # null 表示使用全局默认 Agent
```

### 4.3 FeatureExecution 表变更

新增字段，用于记录执行上下文和性能数据：

```
FeatureExecution
  - agent_id      string (nullable, FK -> AgentConfig.id)  # 新增：执行该阶段的 Agent
  - duration_ms   int (nullable)                           # 新增：阶段执行耗时（毫秒）
  - token_input   int (nullable)                           # 新增：输入 token 数（Agent 支持时记录）
  - token_output  int (nullable)                           # 新增：输出 token 数（Agent 支持时记录）
```

### 4.4 全局配置

系统级默认 Agent 通过环境变量或配置文件指定，不存数据库：

```env
# .env
SOLO100_DEFAULT_AGENT_ID=agent_001
```

优先级（从高到低）：Feature 级别（暂不支持）> Project 级别 > 全局配置 > 系统默认（claude_code）

## 5. 配置管理 UI 变更

### 5.1 Web UI 变更

#### AgentConfig 管理页面（新增）

路径：`/settings/agents`

功能：
- Agent 列表：展示所有已配置的 Agent，标注当前全局默认
- 新增 Agent：选择类型（claude_code / codex），填写名称、API Key、extra_config
- 编辑 Agent：修改名称、extra_config；API Key 只允许重新输入，不回显
- 删除 Agent：若有 Project 引用该 Agent，提示先解除引用
- 设为默认：将指定 Agent 设为全局默认（同时清除其他 Agent 的 is_default）

页面字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| 名称 | 文本 | 显示名称 |
| 类型 | 下拉 | claude_code / codex |
| API Key | 密码框 | 仅写入，不回显 |
| 模型 | 文本 | 对应 extra_config.model |
| 是否默认 | 开关 | 全局默认 Agent |

#### Project 编辑页变更

在 Project 创建/编辑表单中新增：

- "默认 Agent" 下拉选择框，列出所有已配置的 AgentConfig
- 选项包含"使用全局默认"（对应 `default_agent_id = null`）

#### Feature 执行日志变更

在 Feature 详情页的执行日志中，每个阶段新增：
- Agent 名称（如 "Claude Code"）
- 执行耗时（如 "42s"）
- token 消耗（如 "↑1024 ↓2048"，Agent 支持时显示）

#### Agent 性能对比页面（新增）

路径：`/analytics/agents`

功能：
- 按 Agent 类型聚合统计：平均耗时、成功率、token 消耗
- 按阶段（brainstorm/plan/implement）分别展示
- 时间范围筛选（最近 7 天 / 30 天）

后端 API：`GET /api/v1/analytics/agents`，返回按 `agent_type` 聚合的统计数据（平均耗时、成功率、token 消耗），实时查询 `feature_execution` 表。

### 5.2 飞书变更

#### 新增指令

| 指令 | 说明 |
|------|------|
| `/agents` | 列出所有已配置的 Agent |
| `/agent set-default <agent_id>` | 设置全局默认 Agent |
| `/project set-agent <project_id> <agent_id>` | 设置 Project 的默认 Agent |

#### Feature 执行通知变更

在飞书消息卡片的执行状态通知中，新增 Agent 信息行：


## 6. 新增 Agent 接入指南

以下步骤说明如何将一个新的 Agent（例如 `GeminiAgent`）接入 solo100。

### Step 1：实现 BaseAgent

在 `src/agents/` 目录下新建文件 `gemini_agent.py`：

```python
from .base import BaseAgent, Feature, BrainstormResult, Plan, ImplementResult

class GeminiAgent(BaseAgent):

    def brainstorm(self, feature: Feature) -> BrainstormResult:
        # 调用 Gemini API 或 CLI
        prompt = self._build_brainstorm_prompt(feature)
        raw = self._call_gemini(prompt)
        return self._parse_brainstorm(raw)

    def plan(self, feature: Feature, brainstorm: BrainstormResult) -> Plan:
        prompt = self._build_plan_prompt(feature, brainstorm)
        raw = self._call_gemini(prompt)
        return self._parse_plan(raw)

    def implement(self, feature: Feature, plan: Plan) -> ImplementResult:
        prompt = self._build_implement_prompt(feature, plan)
        raw = self._run_gemini_cli(prompt, cwd=feature.worktree_path)
        return self._parse_implement_result(raw, feature.worktree_path)

    def _call_gemini(self, prompt: str) -> str:
        # 实现 Gemini API 调用
        ...

    def _run_gemini_cli(self, prompt: str, cwd: str) -> str:
        # 实现 Gemini CLI 调用（如有）
        ...
```

**注意事项**：
- `implement` 方法必须在 `feature.worktree_path` 内完成代码修改
- 如果 Agent 不自动 commit，`ImplementResult.commit_hash` 留 None，由系统在 implement 完成后自动 commit
- `raw_output` 字段必须填写，用于日志展示和问题排查

### Step 2：注册到 AgentExecutor

在 `src/agents/executor.py` 的 `_instantiate` 方法中注册新类型：

```python
from .gemini_agent import GeminiAgent

def _instantiate(self, config: AgentConfig) -> BaseAgent:
    registry = {
        "claude_code": ClaudeCodeAgent,
        "codex": CodexAgent,
        "gemini": GeminiAgent,  # 新增
    }
    ...
```

### Step 3：更新 AgentConfig 类型枚举

在数据模型层更新 `AgentConfig.type` 的合法值：

```python
# src/models/agent_config.py
AGENT_TYPES = ["claude_code", "codex", "gemini"]  # 新增 gemini
```

### Step 4：更新 Web UI 下拉选项

在 `AgentConfig` 管理页面的类型下拉中新增选项：

```typescript
// frontend/src/components/AgentTypeSelect.tsx
const AGENT_TYPES = [
  { value: "claude_code", label: "Claude Code" },
  { value: "codex",       label: "Codex" },
  { value: "gemini",      label: "Gemini" },  // 新增

```

### Step 5：配置 extra_config schema（可选）

如果新 Agent 有特有配置项，在前端为其定义 schema，用于动态渲染配置表单：

```typescript
// frontend/src/config/agentExtraConfigSchema.ts
export const AGENT_EXTRA_CONFIG_SCHEMA = {
  gemini: [
    { key: "model",       label: "模型",    type: "text",   default: "gemini-2.0-flash" },

```

### Step 6：验证

1. 在 `/settings/agents` 页面新增一个 `gemini` 类型的 AgentConfig
2. 创建或编辑一个 Project，将默认 Agent 设为新建的 Gemini Agent
3. 启动一个 Feature，观察 brainstorm → plan → implement 各阶段是否正常执行
4. 检查 `FeatureExecution` 记录中 `agent_id` 是否正确指向 Gemini Agent

## 7. 与 v0.4 的兼容性说明
