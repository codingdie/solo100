## 1. 版本目标

v0.4 在 v0.3（多 Feature 并行执行）基础上，新增飞书双向集成，使用户可以在手机或飞书桌面端完成所有人工介入操作，与 Web UI 功能完全对等。

**交付标准**：用户可以仅通过飞书 Bot 完成一个 Feature 从创建到 merge 的完整流程，包括所有 Approve / Reject 节点。

---

## 2. 新增功能范围（相对 v0.3 的 delta）

v0.1 ~ v0.3 所有功能保持不变，飞书是新增入口，不替换 Web UI。

### 2.1 新增能力

| 功能 | 说明 |
|------|------|
| 飞书 Bot 创建需求 | 通过 Bot 对话输入需求标题和描述，触发 FeaturePlanner |
| 飞书 Bot 查看 Feature 列表 | 以消息卡片列表形式展示当前所有 Feature 及状态 |
| 飞书 Bot 启动 Feature | 对 pending 状态的 Feature 发起开发流程 |
| 飞书 Bot 人工介入节点操作 | 4 个介入节点（brainstorming 确认、plan 确认、testing 确认、review approve/reject）均可通过卡片按钮操作 |
| 飞书 Bot 查看报告摘要 | 测试报告、code review 报告的摘要通过卡片展示 |
| 飞书状态变更通知推送 | Feature 状态变更、测试失败、review 完成等事件主动推送到飞书群或个人 |

### 2.2 不包含

- 多 Agent 支持（v0.5 实现）
- 飞书替换 Web UI（两者并存，功能对等）
- 飞书内查看完整执行日志（仅摘要，完整日志仍在 Web UI）

## 3. 飞书集成架构

### 3.1 整体架构

```
飞书用户
    │  发送消息 / 点击卡片按钮
    ▼
飞书开放平台
    │  事件订阅（HTTP 回调）
    ▼
solo100 后端
├── Feishu Event Handler（接收事件，解析操作意图）
├── Approval Gateway（处理 approve/reject，驱动状态机）
├── Notification Hub（扩展飞书推送通道）
│       ├── WebSocket → Web UI
│       └── Feishu Bot API → 飞书
└── Feature Executor / FeaturePlanner（业务逻辑不变）
```

### 3.2 飞书 Bot 配置

| 配置项 | 说明 |
|--------|------|
| Bot 类型 | 自建应用（企业自建） |
| 消息权限 | `im:message`、`im:message:send_as_bot` |
| 卡片交互权限 | `im:message.card.action` |
| 事件订阅权限 | `im:message.receive_v1`（接收消息）、`im:message.card.action.trigger`（卡片按钮点击） |
| 推送目标 | 支持群聊（配置通知群）和个人（@用户） |

所需环境变量：

```
FEISHU_APP_ID=cli_xxxxxxxx
FEISHU_APP_SECRET=xxxxxxxx
FEISHU_VERIFICATION_TOKEN=xxxxxxxx   # 事件订阅验证 token
FEISHU_ENCRYPT_KEY=xxxxxxxx          # 可选，事件加密
FEISHU_NOTIFY_CHAT_ID=oc_xxxxxxxx    # 全局通知推送目标群 ID
```

**飞书通知目标架构说明**：

v0.4 废弃 v0.1 的 per-project `feishu_webhook_url` 方案，统一改为全局 `FEISHU_NOTIFY_CHAT_ID` 环境变量。同时支持 per-project 覆盖：`project` 表新增可选字段 `feishu_chat_id TEXT`，若有值则优先使用该群 ID 推送，否则 fallback 到全局 `FEISHU_NOTIFY_CHAT_ID`。

```
Project
  - feishu_chat_id  TEXT (nullable)  # per-project 飞书通知群 ID，覆盖全局配置
```

### 3.3 Webhook 与事件订阅

**事件订阅回调地址**：

```
POST /api/feishu/event
```

飞书开放平台将以下事件推送到此地址：

| 事件类型 | 触发时机 | 处理逻辑 |
|----------|----------|----------|
| `im.message.receive_v1` | 用户向 Bot 发送文本消息 | 解析命令意图，路由到对应处理器 |
| `im.message.card.action.trigger` | 用户点击消息卡片按钮 | 解析 `action_value`，路由到 Approval Gateway |

**URL 验证**：飞书首次配置事件订阅时会发送 `url_verification` 请求，后端需返回 `challenge` 字段完成验证。

**签名验证**：所有事件请求需验证 `X-Lark-Signature` 签名，防止伪造请求。

### 3.4 后端新增模块

| 模块 | 文件位置 | 职责 |
|------|----------|------|
| `FeishuEventHandler` | `app/feishu/event_handler.py` | 接收飞书事件，解析意图，分发到业务层 |
| `FeishuCommandParser` | `app/feishu/command_parser.py` | 解析用户文本消息，识别命令（如 `/list`、`/create`） |
| `FeishuCardBuilder` | `app/feishu/card_builder.py` | 构建各类消息卡片 JSON |
| `FeishuNotifier` | `app/feishu/notifier.py` | 封装飞书 Bot API，发送/更新消息卡片 |

`Notification Hub` 扩展飞书通道（详见第 6 节）。

## 4. 消息卡片设计

所有卡片使用飞书「消息卡片」（Card）格式，支持按钮交互。卡片按钮点击后，后端处理完成再更新卡片内容（`patch` 接口），避免重复操作。

### 4.1 命令入口

用户向 Bot 发送以下文本命令触发对应功能：

| 命令 | 功能 |
|------|------|
| `/list` 或 `列表` | 查看 Feature 列表 |
| `/create` 或 `创建需求` | 进入需求创建流程 |
| `/help` 或 `帮助` | 查看命令列表 |

### 4.2 Feature 列表卡片

触发：用户发送 `/list`

```
┌─────────────────────────────────────┐
│  📋 Feature 列表                     │
│  项目：<project_name>                │
├─────────────────────────────────────┤
│  #001  实现用户登录功能               │
│  状态：🟡 reviewing                  │
│  [查看详情]                          │
├─────────────────────────────────────┤
│  #002  添加数据导出功能               │
│  状态：🔵 implementing               │
│  [查看详情]                          │
├─────────────────────────────────────┤
│  #003  修复分页 Bug                  │
│  状态：⚪ pending                    │
│  [启动开发]                          │
└─────────────────────────────────────┘
```

卡片字段说明：

| 字段 | 说明 |
|------|------|
| Feature 标题 | 截断至 30 字符 |
| 状态 | 带 emoji 的状态标签（见状态 emoji 映射表） |
| 按钮 | pending 状态显示「启动开发」；其他状态显示「查看详情」 |

状态 emoji 映射：

| 状态 | emoji |
|------|-------|
| pending | ⚪ |
| brainstorming / planning / implementing | 🔵 |
| testing | 🟣 |
| reviewing | 🟡 |
| approved / verifying | 🟠 |
| merged | 🟢 |
| failed | 🔴 |
| archived | ⚫ |

### 4.3 创建需求卡片

触发：用户发送 `/create`

Bot 回复引导消息，用户依次发送需求标题和描述（多轮对话），Bot 最终展示确认卡片：

```
┌─────────────────────────────────────┐
│  ✏️ 确认创建需求                      │
├─────────────────────────────────────┤
│  标题：<requirement_title>           │
│  描述：<requirement_description>     │
├─────────────────────────────────────┤
│  [确认创建]          [取消]          │
└─────────────────────────────────────┘
```

确认后，FeaturePlanner 开始拆解，Bot 推送进度通知，完成后推送 Feature 列表卡片（见 4.2）。

「查看详情」按钮跳转到 Web UI 对应 Feature 页面（deep link）。

### 4.6 卡片更新策略

| 场景 | 策略 |
|------|------|
| 用户点击按钮后 | 立即将按钮置为 disabled（防重复点击），后端处理完成后 patch 卡片更新为结果状态 |
| 操作成功 | 卡片底部追加「✅ 操作已确认 by <user_name>」，移除操作按钮 |
| 操作失败 | 卡片底部追加「⚠️ 操作失败：<error_msg>」，保留按钮可重试 |
| 同一节点已被他人操作 | 返回「该节点已被 <user_name> 处理」，卡片更新为已处理状态 |

## 5. 飞书与 Web UI 的状态同步机制

### 5.1 同步原则

- **单一数据源**：Feature 状态以数据库为准，飞书和 Web UI 均从数据库读取，不存在独立状态。
- **操作幂等**：同一介入节点无论从飞书还是 Web UI 操作，结果一致；先到先得，后续操作返回「已处理」。
- **双向实时**：任一端操作后，另一端实时感知（Web UI 通过 WebSocket，飞书通过卡片更新）。

### 5.2 操作流转

```
飞书用户点击卡片按钮
    │
    ▼
Feishu Event Handler 接收 card action 事件
    │  解析 action_value（包含 feature_id、action 类型、操作人）
    ▼
Approval Gateway.handle(feature_id, action, operator)
    │  与 Web UI 操作走同一入口
    ▼
Feature 状态机流转（数据库更新）
    │
    ├──→ Notification Hub
    │         ├── WebSocket 推送 → Web UI 实时刷新
    │         └── Feishu Notifier → 更新飞书卡片 + 推送通知
    │
    └──→ Feature Executor 继续执行下一阶段
```

### 5.3 并发冲突处理

当飞书和 Web UI 同时操作同一介入节点时：

1. Approval Gateway 使用数据库行级锁（`SELECT FOR UPDATE`）保证原子性。
2. 先到的操作成功，状态机流转。
3. 后到的操作检测到状态已变更，返回 `409 Conflict`。
4. Feishu Event Handler 收到 409 后，更新卡片为「已被 <user_name> 处理」。

### 5.4 飞书卡片与 Feature 状态的绑定

每张需要后续更新的卡片，发送后记录 `message_id` 到数据库：

```
FeishuMessage
  - id, feature_id, stage, message_id, chat_id, created_at
  # stage: 对应哪个介入节点的卡片
  # message_id: 飞书消息 ID，用于后续 patch 更新卡片
```

状态变更时，通过 `feature_id + stage` 查找对应 `message_id`，调用飞书 `patch message` 接口更新卡片内容。

## 6. 后端变更（Notification Hub 扩展）

### 6.1 现有 Notification Hub

v0.3 的 Notification Hub 仅支持 WebSocket 推送到 Web UI：

```python
class NotificationHub:
    async def notify(self, event: FeatureEvent) -> None:
        await self.websocket_manager.broadcast(event)
```

### 6.2 v0.4 扩展：多通道推送

扩展为多通道架构，新增飞书通道：

```python
class NotificationHub:
    def __init__(self):
        self.channels: list[NotificationChannel] = [
            WebSocketChannel(),
            FeishuChannel(),   # 新增
        ]

    async def notify(self, event: FeatureEvent) -> None:
        await asyncio.gather(
            *[channel.send(event) for channel in self.channels],
            return_exceptions=True  # 单通道失败不影响其他通道
        )

class NotificationChannel(ABC):
    @abstractmethod
    async def send(self, event: FeatureEvent) -> None: ...

class WebSocketChannel(NotificationChannel):
    async def send(self, event: FeatureEvent) -> None:
        await self.websocket_manager.broadcast(event)

class FeishuChannel(NotificationChannel):
    async def send(self, event: FeatureEvent) -> None:
        card = FeishuCardBuilder.build_notify_card(event)
        await self.feishu_notifier.send_or_update(event, card)
```

### 6.3 FeatureEvent 扩展

新增飞书相关事件类型：

```python
class FeatureEventType(Enum):
    # 已有事件（v0.3）
    STATUS_CHANGED = "status_changed"
    STAGE_COMPLETED = "stage_completed"
    APPROVAL_REQUIRED = "approval_required"
    APPROVAL_DONE = "approval_done"

    # v0.4 新增（飞书专用，不推送到 WebSocket）
    FEISHU_CARD_ACTION = "feishu_card_action"
```

### 6.4 新增 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/feishu/event` | 飞书事件订阅回调（URL 验证 + 事件处理） |

**安全说明**：`/api/feishu/event` 端点加入认证白名单（跳过内部 token 验证），仅通过飞书签名（`X-Lark-Signature`）验证来源合法性。

现有 Approval Gateway API 无需变更，飞书操作复用同一套接口：

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/features/{id}/approve` | Approve（飞书和 Web UI 共用） |
| `POST` | `/api/features/{id}/reject` | Reject（飞书和 Web UI 共用） |
| `POST` | `/api/features/{id}/start` | 启动 Feature 开发（飞书和 Web UI 共用） |

### 6.5 数据库变更

新增 `FeishuMessage` 表（见 5.4），其余数据模型不变。

```sql
CREATE TABLE feishu_messages (
    id          TEXT PRIMARY KEY,  -- UUID
    feature_id  TEXT NOT NULL REFERENCES features(id),
    stage       TEXT NOT NULL,   -- brainstorming | planning | testing | reviewing | notify
    message_id  TEXT NOT NULL,   -- 飞书消息 ID
    chat_id     TEXT NOT NULL,   -- 飞书会话 ID
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_feishu_messages_feature_stage ON feishu_messages(feature_id, stage);
```

## 8. 关键实现注意事项

1. **事件幂等**：飞书事件可能重复投递，`Feishu Event Handler` 需记录已处理的 `event_id`，重复事件直接返回 200 忽略。

2. **卡片 token 有效期**：飞书卡片交互 token 有效期为 30 分钟，超时后按钮点击无效。具体方案：Feature Executor 在每次状态变更时重新推送最新卡片（而非更新旧卡片），确保用户始终有有效的操作入口。

3. **多轮对话状态**：创建需求的多轮对话（标题 → 描述 → 确认）需在后端维护会话状态，可用 Redis 存储 `chat_id → conversation_state`，TTL 设为 10 分钟。会话超时（10 分钟无操作）时，Bot 主动发送超时提示卡片，引导用户重新发起；`conversation_state` 设置 TTL 10 分钟，到期自动清除。

4. **飞书 API 限流**：飞书 Bot API 有频率限制（默认 50 QPS），`FeishuNotifier` 需实现简单的请求队列，避免并发 Feature 状态变更时触发限流。

5. **飞书未配置时降级**：`FEISHU_APP_ID` 未配置时，`FeishuChannel` 静默跳过，不影响 Web UI 正常使用。
