## 1. 版本目标

v0.2 在 v0.1（单 Feature 完整开发流程）基础上，新增**需求管理**能力。

用户可以输入一个大需求（Requirement），由 FeaturePlanner 调用 LLM 自动拆解成 Feature 列表，人工审查调整后，Feature 进入 `pending` 状态等待手动启动开发流程。

**交付标准**：用户能在浏览器里输入一段需求描述，AI 自动拆解出 Feature 列表，人工确认后，每个 Feature 可手动启动 v0.1 的完整开发流程。

---

## 2. 新增功能范围（相对 v0.1 的 delta）

### 新增

| 功能 | 说明 |
|------|------|
| 需求创建 | 用户在 Web UI 输入需求标题 + 描述 |
| FeaturePlanner | 后端服务，调用 LLM 将需求拆解成 Feature 列表 |
| Feature 列表审查 | 人工对 AI 拆解结果进行增删改、调整优先级 |
| Feature 确认入库 | 审查完成后，Feature 批量写入数据库，状态为 `pending` |
| 需求列表页 | 展示所有 Requirement，支持查看关联 Feature |

### 不变（v0.1 原有功能）

- Feature 开发流程状态机（brainstorming → merge）
- 所有人工介入节点（Approve / Reject）
- Claude Code Agent 执行
- Git 工作流（branch、PR、rebase、merge）
- WebSocket 实时推送

### 本版本不包含

- 多 Feature 并行执行
- 飞书集成
- 多 Agent 支持

## 3. 模块设计（新增 / 修改）

### 3.1 FeaturePlanner（新增）

**职责**：接收 Requirement，调用 LLM 拆解成结构化 Feature 列表，返回草稿供人工审查。

**执行流程**：

```
Requirement (title + description)
    ↓
构造 Prompt（见第 7 节）
    ↓
直接调用 Anthropic API（不通过 AgentConfig）
  - API Key：环境变量 ANTHROPIC_API_KEY
  - 模型：claude-opus-4-6（固定，不可配置）
    ↓
解析 LLM 输出 → List[FeatureDraft]
    ↓
持久化为 RequirementFeatureDraft（草稿态，未正式入库）
    ↓
返回草稿列表给前端
```

**FeatureDraft 结构**：

```python
class FeatureDraft:
    title: str           # Feature 标题
    description: str     # Feature 描述（用户故事或功能说明）
    priority: int        # 优先级，1 最高，数字越大越低
    reason: str          # AI 拆解理由（辅助人工判断）
```

**解耦原则**：FeaturePlanner 只负责拆解，不直接创建 Feature 记录。Feature 正式入库由人工确认后的 API 触发。

### 3.2 Approval Gateway（修改）

v0.1 中 Approval Gateway 只处理 Feature 开发流程中的人工介入节点。v0.2 新增一类操作：**Feature 草稿审查确认**。

新增操作类型：
- `confirm_feature_drafts`：人工审查完草稿后，批量确认，将草稿转为正式 Feature（`pending` 状态）

### 3.3 Notification Hub（修改）

新增事件类型：
- `requirement.created`：需求创建成功
- `requirement.planning_started`：FeaturePlanner 开始拆解
- `requirement.planning_done`：拆解完成，草稿就绪，通知前端刷新

## 4. 数据模型变更

### 4.1 新增表：`Requirement`

```
Requirement
  - id: UUID, 主键
  - project_id: FK → Project.id
  - title: str, 需求标题
  - description: text, 需求详细描述
  - status: draft | planning | planning_failed | done
    # draft: 刚创建，尚未触发 FeaturePlanner
    # planning: FeaturePlanner 正在拆解中
    # planning_failed: FeaturePlanner 执行失败（LLM 调用异常或解析失败）
    # done: 拆解完成，草稿已确认，Feature 已入库
  - last_error: TEXT, nullable  # planning_failed 时记录失败原因
  - created_at: datetime
  - updated_at: datetime
```

### 4.2 新增表：`FeatureDraft`

存储 FeaturePlanner 输出的草稿，人工审查期间的临时状态。确认后转为正式 Feature，草稿记录保留（用于溯源）。

```
FeatureDraft
  - id: UUID, 主键
  - requirement_id: FK → Requirement.id
  - title: str
  - description: text
  - priority: int
  - reason: str          # AI 拆解理由
  - status: pending | accepted | rejected
    # pending: 草稿初始状态
    # accepted: 人工确认，已转为正式 Feature
    # rejected: 人工删除
  - feature_id: FK → Feature.id, nullable  # 确认后关联的正式 Feature
  - created_at: datetime
```

### 4.3 变更表：`Feature`

新增字段：

```
Feature
  - requirement_id: FK → Requirement.id, nullable
    # 来自需求管理时有值；手动创建时为 null
  - priority: int, nullable
    # 来自 FeatureDraft 时继承优先级；手动创建时为 null
```

其余字段与 v0.1 保持一致。

## 5. API 设计（新增端点）

基础路径：`/api/v1`

### 5.1 需求管理

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/v1/requirements` | 创建需求 |
| `GET` | `/api/v1/requirements` | 获取需求列表（按 project_id 过滤） |
| `GET` | `/api/v1/requirements/{id}` | 获取需求详情 |
| `POST` | `/api/v1/requirements/{id}/plan` | 触发 FeaturePlanner 拆解 |
| `GET` | `/api/v1/requirements/{id}/drafts` | 获取草稿列表 |
| `PUT` | `/api/v1/requirements/{id}/drafts` | 批量更新草稿（增删改、调整优先级） |
| `POST` | `/api/v1/requirements/{id}/confirm` | 确认草稿，批量创建正式 Feature |

### 5.2 端点详情

**POST /api/v1/requirements**

```json
// Request
{
  "project_id": "uuid",
  "title": "用户认证模块",
  "description": "实现用户注册、登录、JWT 鉴权、密码重置功能，支持邮箱验证"
}

// Response 201
{
  "id": "uuid",
  "status": "draft",
  "created_at": "2026-03-26T10:00:00Z"
}
```

**POST /api/v1/requirements/{id}/plan**

触发 FeaturePlanner 异步执行，立即返回。执行完成后通过 WebSocket 推送 `requirement.planning_done` 事件。

```json
// Response 202
{
  "message": "FeaturePlanner 已启动"
}
```

**PUT /api/v1/requirements/{id}/drafts**

请求体发送完整的草稿列表（含未修改项），服务端做 diff（新增/更新/删除）：
- `id` 为 null：新增草稿（人工手动添加）
- `id` 有值：更新已有草稿
- 不在列表中的已有草稿：视为删除，标记为 `rejected`

```json
// Request
{
  "drafts": [
    {
      "id": "uuid",           // 已有草稿，更新
      "title": "用户注册",
      "description": "实现邮箱注册流程，包含邮箱验证",
      "priority": 1
    },
    {
      "id": null,             // 新增草稿（人工手动添加）
      "title": "第三方登录",
      "description": "支持 Google OAuth2 登录",
      "priority": 3
    }
  ]
}

// Response 200
{
  "drafts": [...]
}
```

**POST /api/v1/requirements/{id}/confirm**

将所有 `pending` 状态的草稿转为正式 Feature（`pending` 状态），`rejected` 的草稿跳过。

```json
// Response 200
{
  "created_features": [
    { "id": "uuid", "title": "用户注册", "status": "pending" },
    { "id": "uuid", "title": "用户登录", "status": "pending" }
  ]
}
```

## 6. 前端页面变更

### 6.1 新增页面

**需求列表页** `/projects/[id]/requirements`

- 展示当前项目下所有 Requirement
- 每条显示：标题、状态（draft / planning / done）、关联 Feature 数量、创建时间
- 操作：新建需求、点击进入需求详情

**需求详情页** `/projects/[id]/requirements/[requirementId]`

分两个阶段展示：

阶段一（status = draft / planning）：
- 展示需求标题和描述
- "开始拆解"按钮，触发 FeaturePlanner
- planning 状态时显示加载动画，WebSocket 监听 `requirement.planning_done`

阶段二（planning_done，草稿审查）：
- 草稿列表，每条显示：标题、描述、优先级、AI 拆解理由
- 支持操作：
  - 编辑标题 / 描述（inline 编辑）
  - 调整优先级（拖拽排序或数字输入）
  - 删除草稿（标记 rejected）
  - 手动新增草稿
- "确认并创建 Feature"按钮，调用 confirm 接口

### 6.2 修改页面

**Feature 列表页** `/projects/[id]/features`（v0.1 已有）

新增展示：
- Feature 卡片上显示来源（来自需求 `{需求标题}` / 手动创建）
- 支持按 Requirement 过滤

**导航**

在项目侧边栏新增"需求"入口，位于"Feature"之上。

### 6.3 新增组件

| 组件 | 说明 |
|------|------|
| `RequirementForm` | 创建需求的表单（标题 + 描述） |
| `FeatureDraftList` | 草稿列表，支持增删改和拖拽排序 |
| `FeatureDraftCard` | 单条草稿卡片，支持 inline 编辑 |
| `PlanningStatus` | FeaturePlanner 执行中的加载状态展示 |

## 7. FeaturePlanner LLM Prompt 设计思路

### 7.1 设计原则

- **输出结构化**：要求 LLM 严格输出 JSON，便于解析，避免自由文本
- **Feature 粒度控制**：每个 Feature 应是独立可交付的功能单元，不能太大（一个 Feature 不超过 3 天工作量），不能太细（不拆到函数级别）
- **保留拆解理由**：每个 Feature 附带 `reason` 字段，帮助人工判断是否合理
- **优先级建议**：AI 给出初始优先级排序，人工可调整

### 7.2 Prompt 结构

```
[System]
你是一个资深软件架构师，擅长将产品需求拆解成可独立开发的 Feature 列表。

拆解规则：
1. 每个 Feature 是一个独立可交付的功能单元
2. Feature 粒度：预估 0.5~3 天工作量，不能更大或更小
3. Feature 之间尽量解耦，减少依赖
4. 优先级从 1 开始，1 最高
5. 严格按 JSON 格式输出，不要输出其他内容

输出格式：
{
  "features": [
    {
      "title": "Feature 标题（简洁，动词开头）",
      "description": "功能描述，说明要实现什么，从用户视角描述",
      "priority": 1,
      "reason": "为什么这样拆，以及优先级的理由"
    }
  ]
}

[User]
项目背景：{project_name}

需求标题：{requirement_title}

需求描述：
{requirement_description}

请将上述需求拆解成 Feature 列表。
```

### 7.3 解析与容错

- 使用 `json.loads` 解析输出，失败时尝试提取 JSON 块（正则匹配 `{...}`）
- 解析失败时记录原始输出，将 Requirement 状态回退到 `draft`，通过 WebSocket 推送错误事件，前端提示用户重试
- 对每个 FeatureDraft 做基础校验：`title` 非空、`priority` 为正整数，不合法的条目跳过并记录警告

### 7.4 异步执行

FeaturePlanner 通过 Celery 任务异步执行，避免阻塞 API 响应。任务完成后：
- 更新 Requirement status 为 `done`（成功）或 `planning_failed`（失败，同时写入 `last_error`）
- 通过 WebSocket 推送 `requirement.planning_done` 事件（成功）或 `requirement.planning_failed` 事件（失败），携带草稿列表或错误信息
