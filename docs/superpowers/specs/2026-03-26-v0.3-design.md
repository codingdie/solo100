# solo100 v0.3 详细设计文档

**日期**: 2026-03-26
**状态**: 草稿
**基于**: [主设计文档](2026-03-25-solo100-design.md)

> **本版本对主设计文档 Feature 状态机的修订**：新增 `rebase_conflict` 状态（介于 `verifying` 和 `merged` 之间，表示 rebase 冲突等待人工处理）。

---

## 1. 版本目标

v0.3 在 v0.2 基础上将执行层从串行升级为并行，实现多个 Feature 同时执行、互不干扰。

**交付标准**：用户可以同时启动多个 Feature 的开发流程，每个 Feature 在独立的 Git worktree 和 branch 中运行，Web UI 可同时监控所有 Feature 的实时进度。

---

## 2. 新增功能范围（相对 v0.2 的 delta）

### 2.1 保持不变

v0.1、v0.2 的所有功能完整保留：
- Web UI 单 Feature 完整开发流程（brainstorming → merge）
- Claude Code Agent 执行
- 需求管理 + FeaturePlanner（AI 拆解 Feature 列表）
- 所有人工介入节点（approve/reject）
- 测试报告、code review 报告

### 2.2 v0.3 新增

| 功能 | 说明 |
|------|------|
| 多 Feature 并行执行 | 多个 Feature 可同时处于 implementing/testing/reviewing 等活跃状态 |
| Celery worker 扩展为多个 | v0.1 已初始化 celery_app（单 worker 串行执行）；v0.3 的实际增量是将 worker 扩展为多个（`--concurrency=N`），每个 Feature 任务在独立 worker 中并行执行 |
| worktree 隔离 | 每个 Feature 在独立 worktree 目录执行，branch 互不干扰 |
| 并行 rebase 策略 | merge 前自动 rebase 到最新主分支，冲突时暂停并通知人工 |
| 多 Feature 监控 UI | Web UI 支持同时展示多个 Feature 的执行进度和日志流 |

### 2.3 不包含

- 飞书双向交互（v0.4）
- 多 Agent 支持（v0.5）

## 3. 并发架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│                    FastAPI Server                    │
│  - REST API（Feature CRUD、人工介入操作）             │
│  - WebSocket（实时推送多 Feature 状态和日志）          │
└──────────────────────┬──────────────────────────────┘
                       │ enqueue task
                       ▼
┌─────────────────────────────────────────────────────┐
│                  Redis（Broker）                     │
│  - 任务队列：feature_tasks                           │
│  - 结果存储：Celery result backend                   │
│  - WebSocket 消息中转：pub/sub channel               │
└──────────────────────┬──────────────────────────────┘
                       │ consume
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Worker 1 │ │ Worker 2 │ │ Worker N │
    │ Feature A│ │ Feature B│ │ Feature C│
    │ worktree │ │ worktree │ │ worktree │
    └──────────┘ └──────────┘ └──────────┘
```

每个 Celery worker 进程同一时刻只处理一个 Feature 的一个阶段任务，通过 worker 数量控制最大并发度。

### 3.2 Celery 任务设计

每个 Feature 的每个阶段对应一个独立 Celery 任务，阶段间串行（前一阶段完成后由 FastAPI 根据状态机决定是否入队下一阶段）。

```python
# tasks/feature_tasks.py

@celery_app.task(bind=True, name="feature.brainstorm")
def task_brainstorm(self, feature_id: int) -> dict:
    """执行 brainstorming 阶段"""
    ...

@celery_app.task(bind=True, name="feature.plan")
def task_plan(self, feature_id: int) -> dict:
    """执行 planning 阶段"""
    ...

@celery_app.task(bind=True, name="feature.implement")
def task_implement(self, feature_id: int) -> dict:
    """执行 implementing 阶段（Agent 写代码）"""
    ...

@celery_app.task(bind=True, name="feature.test")
def task_test(self, feature_id: int) -> dict:
    """执行 testing 阶段"""
    ...

@celery_app.task(bind=True, name="feature.review")
def task_review(self, feature_id: int) -> dict:
    """执行 reviewing 阶段（AI code review）"""
    ...

@celery_app.task(bind=True, name="feature.verify")
def task_verify(self, feature_id: int) -> dict:
    """执行 verifying 阶段（rebase + 测试 + merge）"""
    ...
```

**任务入队时机**：FastAPI 的 `ApprovalGateway` 在处理人工介入操作后，根据状态机流转结果决定是否入队下一阶段任务。自动流转阶段（如 brainstorming → planning）由任务本身完成后回调 FastAPI 触发入队。

### 3.3 Worker 配置

```python
# celery_config.py

CELERY_CONFIG = {
    "broker_url": "redis://localhost:6379/0",
    "result_backend": "redis://localhost:6379/1",
    "task_serializer": "json",
    "result_serializer": "json",
    "accept_content": ["json"],
    # 每个 worker 同时只处理 1 个任务，避免单 worker 内并发导致 worktree 冲突
    "worker_concurrency": 1,
    # 任务超时：Agent 执行最长 30 分钟
    "task_soft_time_limit": 1800,
    "task_time_limit": 2100,
    # 任务失败重试策略（网络抖动等临时错误）
    "task_max_retries": 2,
    "task_default_retry_delay": 30,
}
```

**worker 数量**：通过 Docker Compose `replicas` 控制，默认 3 个 worker，即最多 3 个 Feature 同时执行。可按需调整。

### 3.4 worktree 隔离

每个 Feature 在独立目录执行，目录结构：

```
/data/worktrees/
  ├── feature-{id}-{slug}/     # Feature A 的 worktree
  │     └── <repo files>
  ├── feature-{id}-{slug}/     # Feature B 的 worktree
  │     └── <repo files>
  └── ...
```

**worktree 生命周期**：

```
Feature 启动（brainstorming 入队）
    ↓
GitManager.create_worktree(feature_id)
  - git worktree add /data/worktrees/feature-{id}-{slug} -b feat/{id}-{slug}
    ↓
各阶段任务在该 worktree 目录内执行
    ↓
Feature merged 或 archived
    ↓
GitManager.cleanup_worktree(feature_id)
  - git worktree remove /data/worktrees/feature-{id}-{slug}
  - git branch -d feat/{id}-{slug}（远程 branch 由 PR merge 时自动删除）
```

**并发安全**：每个 worktree 目录独立，不同 Feature 的 Agent 操作互不影响。主仓库目录（bare clone）只用于 `git worktree add`，不直接在其中执行 Agent 任务。

### 3.5 日志流转发

Agent 执行过程中的实时日志通过 Redis pub/sub 转发到 WebSocket：

```
Worker（Agent 执行）
    ↓ redis.publish("feature:{id}:log", log_line)
Redis pub/sub
    ↓ subscribe
FastAPI WebSocket Handler
    ↓ ws.send(log_line)
前端 WebSocket 客户端
```

## 4. 并发冲突处理策略

### 4.1 rebase 冲突检测

verification 阶段执行 rebase 时，`GitManager` 捕获冲突并上报：

```python
class GitManager:
    def rebase_onto_main(self, feature_id: int) -> RebaseResult:
        worktree = self._get_worktree_path(feature_id)
        repo = Repo(worktree)

        try:
            repo.git.fetch("origin", self._main_branch)
            repo.git.rebase(f"origin/{self._main_branch}")
            return RebaseResult(success=True)

        except GitCommandError as e:
            if "CONFLICT" in e.stderr or repo.is_dirty():
                # 中止 rebase，保留冲突现场供人工查看
                repo.git.rebase("--abort")
                conflicts = self._extract_conflict_files(e.stderr)
                return RebaseResult(success=False, conflicts=conflicts)
            raise
```

### 4.2 冲突时的状态流转

```
verifying
    ↓ rebase 冲突
Feature.status → rebase_conflict（新增状态）
    ↓
NotificationHub 推送通知（Web UI）
    ↓ 人工解决冲突后，在 Web UI 点击"冲突已解决，继续"
Feature.status → verifying（重新触发 verify 任务）
```

**新增状态 `rebase_conflict`**：介于 `approved` 和 `verifying` 之间，表示 rebase 冲突等待人工处理。人工有两个选项：
- **手动解决后继续**：在本地 worktree 解决冲突，push 后点击"继续"，重新触发 verify
- **放弃本次 verification**：回退到 `approved` 状态，等待下次手动触发

### 4.3 人工介入流程

```
1. Web UI 收到 rebase_conflict 通知
   - Feature 卡片显示"Rebase 冲突"红色标记
   - 展示冲突文件列表
   - 提示：worktree 路径 /data/worktrees/feature-{id}-{slug}

2. 人工在本地终端操作：
   cd /data/worktrees/feature-{id}-{slug}
   git fetch origin
   git rebase origin/main
   # 手动解决冲突
   git add .
   git rebase --continue
   git push --force-with-lease origin feat/{id}-{slug}

3. 人工在 Web UI 点击"冲突已解决，重新验证"
   → POST /api/v1/features/{id}/resolve-conflict
   → Feature.status → verifying
   → 重新入队 task_verify
```

### 4.4 多 Feature 并行时的 rebase 顺序

多个 Feature 同时进入 verifying 时，rebase 顺序不做强制排队，各自独立 rebase 到当前最新主分支。先 merge 的 Feature 会使后续 Feature 的 rebase 基准更新，但这是正常的 Git 工作流，不需要特殊处理。

## 5. 数据模型变更

### 5.1 Feature 状态扩展

在原有状态机基础上新增 `rebase_conflict` 状态：

```
原状态集合（v0.2）：
  pending | brainstorming | planning | implementing | testing |
  reviewing | approved | verifying | merged | failed | archived

v0.3 新增：
  rebase_conflict  # rebase 冲突，等待人工解决
```

**完整状态机（v0.3）**：

```
pending → brainstorming → planning → implementing → testing → reviewing
    → approved → verifying → merged
                          ↘ rebase_conflict → verifying（人工解决后）
                                           → approved（人工放弃）
```

### 5.2 Feature 表新增字段

```
Feature
  + celery_task_id TEXT  # 当前阶段对应的 Celery task ID，任务完成后清空（设为 NULL）
  + worktree_path        # worktree 目录路径（v0.1 已有，v0.3 确保并发场景下正确赋值）
```

注：`conflict_files` 不存储在 `Feature` 表，仅保留在 `FeatureExecution.result_json` 中（见 5.3 节），避免冗余。

### 5.3 FeatureExecution 表新增阶段

`stage` 枚举新增 `rebase_conflict` 记录冲突事件：

```
stage 枚举（v0.3）：
  brainstorming | planning | implementing | testing | reviewing | verifying | rebase_conflict
```

`rebase_conflict` 类型的执行记录 `result_json` 结构：

```json
{
  "conflict_files": ["src/foo.py", "src/bar.py"],
  "rebase_base_commit": "abc123",
  "detected_at": "2026-03-26T10:00:00Z",
  "resolved_at": null,
  "resolution": null
}
```

## 6. 前端变更

### 6.1 多 Feature 并行监控 UI

**Feature 列表页**（主视图）改为支持多 Feature 同时展示活跃状态：

```
┌─────────────────────────────────────────────────────────────┐
│  Features                                    [+ New Feature] │
├─────────────────────────────────────────────────────────────┤
│  ● Feature A  [implementing]  ████████░░  80%   [查看详情]   │
│  ● Feature B  [testing]       ██████░░░░  60%   [查看详情]   │
│  ⚠ Feature C  [rebase_conflict]  冲突文件: 2    [处理冲突]   │
│  ○ Feature D  [pending]                         [启动]       │
└─────────────────────────────────────────────────────────────┘
```

- 活跃 Feature（非 pending/merged/archived）显示实时进度条和当前阶段
- `rebase_conflict` 状态显示警告图标和冲突文件数，提供"处理冲突"快捷入口
- 列表自动刷新（WebSocket 驱动，无需手动刷新）

### 6.2 Feature 详情页

详情页支持多 Feature 同时打开（浏览器多标签），每个详情页独立订阅对应 Feature 的 WebSocket 日志流。

**rebase_conflict 状态的详情页**新增冲突处理面板：

```
┌─────────────────────────────────────────────────────────────┐
│  ⚠ Rebase 冲突                                              │
│                                                             │
│  以下文件存在冲突，请在本地 worktree 手动解决：              │
│    • src/foo.py                                             │
│    • src/bar.py                                             │
│                                                             │
│  worktree 路径：/data/worktrees/feature-42-add-login        │
│                                                             │
│  解决步骤：                                                  │
│    1. cd /data/worktrees/feature-42-add-login               │
│    2. git rebase origin/main                                │
│    3. 解决冲突后 git rebase --continue                      │
│    4. git push --force-with-lease                           │
│                                                             │
│  [冲突已解决，重新验证]          [放弃，回到 approved]       │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 WebSocket 多路复用

v0.2 的 WebSocket 连接为单 Feature 设计，v0.3 升级为支持多 Feature 订阅：

**迁移路径**：
- v0.1/v0.2 的 `WS /api/v1/ws/features/{feature_id}` 保留兼容，详情页继续可用
- v0.3 新增 `WS /api/v1/ws/features/status`，供列表页订阅多 Feature 状态流

**连接方式**：

```
# 订阅单个 Feature 的日志（详情页使用，v0.1/v0.2 兼容端点）
WS /api/v1/ws/features/{feature_id}

# 订阅所有 Feature 的状态变更（列表页使用，v0.3 新增）
WS /api/v1/ws/features/status
```

**状态变更消息格式**：

```json
{
  "type": "feature_status_changed",
  "feature_id": 42,
  "old_status": "verifying",
  "new_status": "rebase_conflict",
  "conflict_files": ["src/foo.py"],
  "timestamp": "2026-03-26T10:00:00Z"
}
```

**日志消息格式**（与 v0.2 兼容）：

```json
{
  "type": "log",
  "feature_id": 42,
  "stage": "implementing",
  "line": "[Agent] Writing src/foo.py...",
  "timestamp": "2026-03-26T10:00:00Z"
}
```

### 6.4 新增 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/features/{id}/resolve-conflict` | 人工解决冲突后触发重新验证 |
| POST | `/api/v1/features/{id}/abort-conflict` | 放弃冲突处理，回退到 approved |
| GET  | `/api/v1/features/{id}/worktree-info` | 获取 worktree 路径和冲突文件信息 |

## 7. 部署变更

### 7.1 Docker Compose 新增服务

```yaml
services:
  api:
    # 不变

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

  celery_worker:
    build: .
    command: celery -A app.celery worker --loglevel=info --concurrency=1
    deploy:
      replicas: 3          # 最多 3 个 Feature 并行，按需调整
    volumes:
      - worktrees:/data/worktrees
      - ssh_keys:/root/.ssh:ro
    depends_on:
      - redis

  frontend:
    # 不变

volumes:
  redis_data:
  worktrees:               # 新增：worktree 持久化卷
  ssh_keys:
```

### 7.2 worktree 卷挂载

`worktrees` 卷需要同时挂载到 `api` 容器和 `celery_worker` 容器，确保：
- API 容器可以读取 worktree 路径信息（用于展示给用户）
- Worker 容器在该卷内执行 git worktree 操作

## 8. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| worker 并发模式 | `--concurrency=1`（每 worker 单任务）| Agent 任务 I/O 密集但有状态，单任务避免 worktree 竞争 |
| rebase 冲突处理 | 中止 rebase，人工在本地解决 | 冲突解决需要语义理解，不适合自动化 |
| 多 Feature 状态推送 | 独立 WebSocket channel | 列表页和详情页订阅不同粒度，减少不必要的数据传输 |
| worktree 存储 | Docker volume 持久化 | 容器重启后 worktree 不丢失，避免重新 clone |
| rebase 顺序 | 不强制排队 | 简化实现，符合标准 Git 工作流，冲突概率低 |
